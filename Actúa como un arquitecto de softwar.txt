========================
2. HALLAZGOS TÉCNICOS YA CONFIRMADOS
========================

Estos puntos ya fueron validados en ejecución real y deben tomarse como hechos confirmados del entorno actual:

2.1. Host y puerto del HSM
- Host probado: 192.168.1.204
- Puerto probado: 2003

2.2. Formato de trama que sí funcionó
La conexión funcional actual usa este framing:

- 2 bytes iniciales con la longitud del payload
- esos 2 bytes van en big-endian
- después de esos 2 bytes viene el payload
- el payload se envía como ASCII

Es decir:

[Length: 2 bytes big-endian] + [Payload ASCII]

2.3. Casos que respondieron correctamente
Se hicieron pruebas de descubrimiento con varios formatos. Los dos que sí respondieron fueron:

Caso 1:
- Header = "PINV"
- Command = "AS"

Caso 2:
- Header = "0000"
- Command = "0000"

Por ahora, el perfil principal usado para conexión es el primero:
- Header = "PINV"
- Command = "AS"

2.4. Caso que NO quiero repetir
Ya no quiero volver a hacer lógica de testCases ni discovery dinámico en controllers.
Eso ya se hizo y ya cumplió su objetivo.
No quiero volver a endpoints tipo “descubrir formato”.
Quiero seguir desde la arquitectura limpia actual.

2.5. Naturaleza de los datos
- La longitud va en binario, específicamente 2 bytes big-endian
- El payload se construye como texto ASCII
- La respuesta del servidor también se recibe con:
  - 2 bytes iniciales de longitud en big-endian
  - luego un payload
- El payload recibido puede representarse de dos formas:
  1. como ASCII, para lectura humana si el contenido es textual
  2. como HEX, para diagnóstico y trazabilidad técnica

2.6. Conversión de respuesta a lenguaje normal
Para interpretar una respuesta:
- primero leer los 2 bytes de longitud como UInt16 big-endian
- luego leer esa cantidad de bytes del payload
- luego convertir el payload a:
  - ASCII: Encoding.ASCII.GetString(payloadBytes)
  - HEX: BitConverter.ToString(payloadBytes).Replace("-", "")
- si el payload contiene texto legible, mostrarlo como ASCII
- si contiene datos no legibles o mezcla de caracteres, conservar también su representación HEX
- por ahora, el sistema devuelve ambos: ASCII y HEX

========================
3. DECISIÓN DE ARQUITECTURA ACTUAL
========================

La arquitectura actual ya fue reorganizada con estas ideas:

3.1. La API no debe conocer detalles del protocolo
No quiero que los controllers conozcan:
- el Header real del protocolo
- el Command real del protocolo
- framing TCP
- bytes
- sockets

3.2. Los detalles del protocolo deben quedar encapsulados
Para eso se usó una abstracción por perfiles de protocolo.

3.3. Se implementó una arquitectura genérica basada en:
- transporte TCP
- builder/parser de trama
- perfiles de protocolo
- ejecutor genérico de comandos
- servicio/probe de conexión

========================
4. DISEÑO ACTUAL DE COMPONENTES
========================

Estos son los conceptos importantes del diseño actual:

4.1. IHsmTransport
Responsabilidad:
- enviar y recibir bytes por TCP
- no debe conocer lógica HTTP ni casos de uso

4.2. HsmRequestFrameBuilder
Responsabilidad:
- construir la trama
- recibe una definición de comando
- genera:
  [2 bytes big-endian de longitud] + [payload ASCII]

4.3. HsmResponseFrameParser
Responsabilidad:
- leer una respuesta raw
- interpretar los primeros 2 bytes como longitud big-endian
- extraer payload
- producir:
  - longitud
  - ASCII
  - HEX
  - resumen

4.4. IHsmProtocolProfile
Responsabilidad:
- encapsular cómo se construye el comando de conexión o futuros comandos según el perfil activo

4.5. LegacyPinvProtocolProfile
Perfil actualmente principal
Debe devolver como probe de conexión:
- Header = "PINV"
- Command = "AS"

4.6. ZeroEchoProtocolProfile
Perfil alternativo probado
Debe devolver como probe de conexión:
- Header = "0000"
- Command = "0000"

4.7. HsmProtocolProfileFactory
Responsabilidad:
- elegir el perfil activo desde configuración
- por ejemplo:
  - LegacyPinv
  - ZeroEcho

4.8. IHsmCommandExecutor
Responsabilidad:
- recibir una definición de comando
- construir la trama
- enviarla por transporte
- parsear la respuesta
- devolver una respuesta parseada genérica

4.9. IHsmConnectionProbe
Responsabilidad:
- ejecutar la verificación de conexión usando el perfil activo
- medir latencia
- devolver un resultado limpio para la API

========================
5. MODELOS IMPORTANTES
========================

Estos son los modelos conceptuales que ya se definieron:

5.1. HsmCommandDefinition
Representa un comando interno, algo como:
- Header
- Command
- Data opcional

Este modelo es interno, NO debe exponerse como DTO público HTTP.

5.2. HsmParsedResponse
Representa una respuesta parseada, con:
- Length
- PayloadAscii
- PayloadHex
- Summary

También es interno.

5.3. HsmConnectionStatus
Representa la respuesta de negocio para el endpoint de conexión, con algo como:
- IsConnected
- ProfileName
- Endpoint
- LatencyMs
- Summary
- ResponseAscii
- ResponseHex

Este sí puede mapearse al DTO HTTP.
